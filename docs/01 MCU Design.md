# MCU Design

The MCU handles handles voice lifecycle, parameters, MIDI parsing and generation of control rate signals.

#### Software Architecture

The MCU uses a bare-metal, super-loop approach with interrupts.

![Software Architecture](assets/images/sw_architecture.svg)


| # | Notes | 
|---|-------|
| 1 | The bare-metal super loop handles control rate signals, patch management, MIDI message parsing and voice lifecycle|
| 2 | Incoming serial MIDI is stored in a ring-buffer for processing by the super-loop.|
| 3 | The audio ISR indicates the point at which latched events can be written to voice ram (VRAM). |
| 4 | MIDI bytes are received by the MIDI module which interrupts the CPU to store the byte|
| 5 | Parameters and control rate signals are latched into VRAM with access controlled via the VRCR peripheral|
| 6 | The I2S peripheral drives timing through sample request and audio IRQ. It generates stereo audio output|

The interrupts and shared voice RAM decouples the MCU from the audio pipeline and either can be built and tested independently.

There is no DAE or audio DSP in the MCU (firmware) as sample generation and output is entirely the concern of the audio pipeline. 

#### MCU Components

The MCU is built around the [Yosys PicoRV32](https://github.com/YosysHQ/picorv32) soft-core processor with a set of solution specific MMIO modules:

| Module | Purpose |
| ------ | ------- |
| SRAM   | CPU SRAM (BRAM) |
| VRAM   | Shared Voice RAM |
| VRCR   | VRAM control register |
| TRACE  | Serial-tx module (used for debug/trace) |
| MIDI   | Serial-rx module (used for MIDI in) |
| GPO    | On-board LEDs and misc IO|


## Fixed Point Types

Much of the synthesiser code in the firmware is ported from my ARM MCU based projects which have an FPU and therefore use single precision, normalised, floating point. 

This MCU has no FPU, floating point support or math libraries so fixed point decimal is used.

| Signal | Range | Type | Format | Notes |
| ------ | ----- | ------ | ----- | ---- |
| Phase Acc | 0-2^24 | uint32_t | Q24.0 | Counter - no fraction |
| FCW  | 0-2^24 | uint32_t | Q24.0 | Derivative of Phase Acc width |
| Control Rate Signals   | Q1.15| int32_t | SQ1.15 | Bipolar normalised (-1.0:~1.0) |
| User Parameters | 0-2^7 | uint8_t |-  | MIDI CC (7-bit) values |

Rather than create fixed point math functions (or approximations) I use Python scripts to generate precomputed lookup tables. While these discrete values will change the characteristics of the synthesiser subtly, musically these are irrelevant.

Parameters are set using MIDI CC messages. They contain either a direct value or an indirect index into a LUT.

The LUTs are generated by the Python script ```gen_luts.py```.  All are mapped to 7-bit MIDI range (0-127 index value)

## VRAM Boundary

The VRAM (Voice RAM) is the interface between the control plane and the audio pipeline.

**The MCU is only a signal source, not a signal processor.**

It computes control-rate modulator outputs (amp envelope, mod envelope and lFO levels) and writes normalised Q1.15 values to VRAM once per control block.

A control block is 46 samples in length, ~1ms at 46875Hz.

It does not apply these directly to their destinations but instead writes them to a shared VRAM.  Voice lifecycle events (note-on, off, stealing) and other static configuration changes (waveform select, FCW, filter settings etc) are also writen to VRAM but only on a change event, not every block.

**The pipeline runs all DSP.**

At a block boundary the pipeline reads values and configuration from VRAM and computes modulated values,  pitch, filter cutoff, VCA level etc before running the sample-rate signal chain. 

Depth scaling, any non-linear shaping of modulator outputs, and per-sample
interpolation between block values are all pipeline responsibilities.

Pre-multiplying each modulator output by its destination depth on the MCU might
save a small number of pipeline multiply operations but would also move part of the DSP signal decisions into firmware breaking the clean boundary.

The pipeline's hardware 18×18 DSP multipliers are also useful for these multiplications and the values have been sized to infer use of these.
